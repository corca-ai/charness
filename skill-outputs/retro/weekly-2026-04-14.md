## Waste

- We repeatedly treated adjacent surfaces as if they guaranteed each other:
  source checkout vs managed checkout vs installed PATH binary vs checked-in
  plugin export vs host cache. That caused stale-binary, export-drift, and
  stale-lock failures to surface late.
- We still let some generated or vendored surfaces reach standing validation
  before their policy was explicit. The `specdown` markdown burst and the
  late-discovered plugin export drift were both examples of "ship the surface,
  then discover the gate semantics afterward."
- We sometimes verified the right thing too late or in the wrong order. Running
  sync/closeout concurrently and only then discovering plugin-tree corruption is
  a process smell, not just a one-off mistake.
- We were too loose about discovery semantics while tightening support-tool
  flows. "Support-backed", "integration-only", "host-visible", and
  "repo-local support surface" were distinct ideas in code, but not always in
  our working mental model.
- We still depended too much on live human interpretation of machine state
  before turning it into structured next steps. The `doctor cautilus` follow-on
  work existed because the first pass left too much implied.

## Critical Decisions

- We chose to make generated support for tools such as `cautilus` discoverable
  through `find-skills`, discovery stubs, and `doctor` guidance instead of
  forcing every support surface to become a top-level host-visible skill. That
  kept the architecture cleaner, but only after we made the discovery path
  explicit.
- We chose to make install guidance more structural by requiring install
  entrypoint URLs in support-backed manifests instead of hard-coding every exact
  install command into `doctor`.
- We chose to turn verifier selection into a repo-owned helper rather than keep
  it as tribal memory, which was the right direction because many slices were
  now touching multiple validator families and generated surfaces.
- We chose to narrow the old update-propagation proof from a standing concern to
  an on-demand regression fixture. That was correct, but it also showed that we
  need a more disciplined habit of removing solved worries from active working
  memory sooner.

## Expert Counterfactuals

- Gary Klein: run a premortem on every "obvious" surface assumption before
  acting. The question would have been: "If this update/install/support change
  fails in a way that embarrasses us later, which stale surface is most likely
  to be the real culprit?" That would have pushed source, managed, installed,
  and exported surfaces into separate explicit checks earlier.
- John Ousterhout: treat repeated surface-sync pain as a design smell, not an
  operational nuisance. The better move is to reduce independent state surfaces
  or make one derive mechanically from another, instead of relying on humans to
  remember every sync edge.

## Next Improvements

- `workflow`: keep a default review checklist for any install/update/support
  change that explicitly covers source checkout, managed checkout, installed
  CLI, checked-in plugin export, host cache, and PATH visibility as separate
  seams.
- `workflow`: do not run sync-producing commands in parallel with closeout or
  packaging verification. Generate first, verify second.
- `capability`: keep pushing "next action" into structured output. Any
  `doctor`/`install`/`sync-support` surface that still requires prose
  interpretation should be treated as incomplete.
- `capability`: add one repo-owned validator for vendored-surface policy so
  markdown or packaging exceptions are declared at intake time instead of
  discovered at standing-quality time.
- `capability`: consider a surface-separation helper or checklist for support
  discoverability so "host-visible", "repo-local generated", and
  "integration-only" cannot be conflated while implementing follow-ons.
- `memory`: keep major incident families in durable debug artifacts, but also
  summarize the cross-cutting lesson in the weekly retro so the repo remembers
  patterns, not only isolated incidents.
- `memory`: when a concern is fully resolved, demote it from handoff/current
  focus to on-demand regression status quickly. Resolved worries should stop
  consuming active planning bandwidth.
