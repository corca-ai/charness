# Rejected Alternatives

For a risky or design-shaping contract, keep a short `Deliberately Not Doing`
section in the durable spec when future readers are likely to reopen the same
branch. This is where the important rejected alternatives belong.

Good triggers:

- deleting or renaming an established surface
- choosing one canonical implementation and retiring another
- refusing a speculative compatibility path
- declining a "keep both for safety" argument that would create real ongoing
  maintenance cost

Keep it short:

- alternative considered
- why it was rejected
- when the team would revisit it

This should not become a full appendix of every brainstormed idea.
Record only the rejected paths that a future maintainer would otherwise
re-litigate.
