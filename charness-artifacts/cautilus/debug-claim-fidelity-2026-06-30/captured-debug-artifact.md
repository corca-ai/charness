# Non-gitignore-aware repo file scanners (CI-only over-inclusion) Debug
Date: 2026-06-30

## Problem

A repo file scanner walks the filesystem and includes gitignored files, so it
passes locally but fails only in CI: a quality/validation gate is green locally and
red in CI for the same commit because the scanner's notion of "files in the repo"
diverges from git's. Symptom class: "로컬은 통과하는데 CI에서만 실패" — local pass,
CI-only fail, gitignored files included.

## Correct Behavior

A scanner asking "what files are in this repo?" must use git's gitignore-aware view
(`git ls-files --cached --others --exclude-standard`, i.e.
`scripts.repo_file_listing.iter_repo_files`/`iter_matching_repo_files`), so the
candidate set is identical on a clean local checkout and in CI regardless of which
gitignored artifacts are present. Restores: deterministic, environment-independent
gate results.

## Observed Facts

- Canonical gitignore-aware seam exists: `scripts/repo_file_listing.py:22,50,85`
  (`git ls-files -z --cached --others --exclude-standard`).
- 14 scanners in `scripts/*.py` reconstruct repo membership with raw
  `rglob`/`glob`/`os.walk` instead (see Sibling Search), each with its own
  hand-maintained denylist (`operator_acceptance_lib.py:7 EXCLUDED_DIRS`,
  `narrative_adapter_lib.py:64`, `list_external_links.py:12`) — a partial, drifting
  copy of `.gitignore`.
- `.gitignore` includes CI-generated dirs absent from those denylists: `reports/`,
  `mutants/`, `pytest-tmp/`, `.artifacts/`, `skills/support/generated/*`,
  `.coverage*` — generated by CI quality/mutation jobs; a dev skipping those steps
  lacks them locally.
- Gate IS wired CI-failing (`run-quality.sh:500`: `inventory_gitignore_scan_hygiene.py
  --require-empty --require-git-file-listing`) yet reports 0 findings (Detection Gap).
- Falsifier on "git missing in CI": `.github/workflows/quality-core.yml:98-100` uses
  `actions/checkout@v6` `fetch-depth: 0` — full git in CI; mechanism works WITH git.

## Reproduction

CI-faithful (full git checkout) against real `operator_acceptance_lib._iter_markdown_files`:

```
tmp git repo, full checkout: docs/acceptance.md (tracked) + reports/generated.md
  (gitignored via .gitignore `reports/`), both "## functional check"
git ls-files -> ['.gitignore','docs/acceptance.md'];  git check-ignore reports/generated.md -> rc 0
'reports' in EXCLUDED_DIRS -> False
_iter_markdown_files(repo)  -> ['docs/acceptance.md','reports/generated.md']  # ignored file INCLUDED
```

Local pass / CI fail: locally `reports/` does not exist -> only the tracked file
scanned -> pass. In CI a prior step generates `reports/` -> the gitignored file is
scanned -> spurious finding -> red. Same delta for `list_external_links.py` when git
is unusable (its `git ls-files` path falls back to `repo_root.rglob('*')`).

## Candidate Causes

- (control-flow) Raw `rglob`/`os.walk` never consults git -> always includes
  gitignored files. [confirmed primary]
- (env) git unavailable in CI -> seam falls back to rglob. [falsified: quality-core
  uses fetch-depth 0; latent only for `list_external_links` fallback]
- (data drift) Hand-rolled denylist is a partial copy of `.gitignore` omitting
  CI-generated dirs. [confirmed — live mechanism with git present]
- (detection) Hygiene gate fails closed but its scope misses these files.
  [confirmed — Detection Gap]

## Hypothesis

Falsifiable: "raw `rglob` + denylist (not the git-aware seam) scans a gitignored
file under a denylist-omitted directory even with full git present." Predicted:
adding such a file includes it; routing through `iter_repo_files` excludes it.

## Verification

Confirmed by the reproduction: `_iter_markdown_files` includes gitignored
`reports/generated.md` with `git ls-files`/`git check-ignore` functional, because
`reports` is not in `EXCLUDED_DIRS`. The seam excludes it by construction
(`--others --exclude-standard` never emits ignored paths).

## Root Cause

No single enforced gitignore-aware listing seam. The correct seam exists
(`repo_file_listing.iter_repo_files`) with a positive exemplar
(`scripts/lint_ignore_inventory_lib.py:9,56` — git-aware list first, denylist only
as a secondary belt). But 14 sibling scanners bypass it and reconstruct repo
membership with raw `rglob`/`glob`/`os.walk` plus a bespoke denylist that is a
partial, drifting copy of `.gitignore`. The candidate set diverges from git's view
exactly when a gitignored path the denylist omits is present at scan time — a
CI-characteristic condition (build/test/mutation generate `reports/`, `mutants/`,
`pytest-tmp/`, coverage), not a local-incremental-run condition. The defect is
structural (missing seam-adoption invariant), not a per-scanner typo.

## Invariant Proof

- Invariant: every repo-membership enumeration uses the git-aware seam, so
  candidate(local) == candidate(CI). Producer = file-listing helper; final consumer
  = each gate's pass/fail.
- Producer Proof: `repo_file_listing.git_list_repo_files` never emits ignored paths
  (repro: seam excludes `reports/generated.md`).
- Final-Consumer Proof: `operator_acceptance_lib._iter_markdown_files` bypasses the
  producer and emits the ignored path -> result diverges (consumer never calls producer).
- Interface-Shape Sibling Scan: the producer-bypass shape recurs in 14 consumers.
- Non-Claims: I did NOT read a specific failing CI log; the CI-only failure is
  reconstructed locally by materializing the generated-file condition.

## Prevention

1. Converge the 14 consumers onto `repo_file_listing.iter_repo_files`/
   `iter_matching_repo_files` (the `lint_ignore_inventory_lib` pattern); drop the
   bespoke denylists or demote to a secondary belt over git-filtered paths.
2. Broaden the hygiene gate: add `scripts/*.py`/`scripts/**/*.py`/`skills/**/*.py`
   to its globs, broaden the receiver heuristic, scope suppression to the branch.
3. Add a repo-level test running the gate over `scripts/*.py` asserting findings==0
   so the scope cannot silently narrow again.
