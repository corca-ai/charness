## Resolution Brief — corca-ai/charness#161

**Classification**: feature

**Reporter JTBD**: before a bounded fresh-eye critique runs, give the
reviewer a compact deterministic packet of surface, adapter, doc-link,
generated/derived/audit-only/rewrite-role, and test-coverage signals so
critique focuses on judgment instead of rediscovering enumerable
inventory across multiple passes.

**Why now**: Cautilus repo already shipped a repo-local
`scripts/prepare-surface-critique-packet.mjs` that produces a
`cautilus.surface_critique_packet.v1` payload for its `release-packaging`
surface. The lesson is clear (each fresh-eye pass without deterministic
context surfaces one more inventory miss), and the boundary is also
clear (Cautilus owns its surface specifics; Charness should own the
portable packet shape, the adapter hook points, and the workflow
requirement that fresh-eye critique consumes the packet first).

**Proposed boundary**:

- in scope:
  - portable packet schema (envelope + named sections) under a Charness
    namespace (e.g. `charness.critique_prepare_packet.v1`)
  - adapter hook contract that lets `.agents/critique-adapter.yaml`
    declare which packet sections apply, and how each section's content
    is produced (static config vs adapter-resolved scanner script)
  - a Charness-owned `prepare` script (or skill helper) that reads the
    adapter, executes declared section producers, and emits the packet
    as a deterministic JSON+markdown artifact
  - workflow requirement on the `critique` skill that fresh-eye
    reviewers consume the packet before broad repo sampling, with one
    durable closeout field that records which packet was used
  - one Charness-owned default section ("changed files + owning skill
    surfaces" using existing `check_changed_surfaces.py` style data) so
    the contract has a concrete consumer in this repo

- out of scope:
  - migrating cautilus's existing
    `prepare-surface-critique-packet.mjs` into charness; the migration
    happens cautilus-side once the contract exists and is documented
  - prescribing the exact content of every candidate section (release
    control docs, audit-only marketplace metadata, etc. are
    consumer-specific; charness names the contract surface, not the
    content)
  - retro-prepare counterpart (issue mentions `critique prepare` /
    `retro prepare`; defer retro hook to a follow-up slice once
    critique side is validated by at least one repo)

**Non-goals**:

- making Charness aware of any cautilus-specific role classification
  (`source` / `derived` / `audit-only` / `rewrite`); roles stay
  consumer-defined
- enforcing packet *content correctness* — Charness validates packet
  *shape* and *presence*; correctness of section findings stays the
  scanner author's responsibility
- collapsing critique into a "scanner-only" gate; the packet feeds
  judgment, not replaces it

**Acceptance checks** (subject to spec slice):

- a `charness.critique_prepare_packet.v1` schema lands as a
  shared reference with a JSON Schema or equivalent contract validator
- `.agents/critique-adapter.yaml` (new adapter) supports declaring
  `packet_sections` with at least: section id, title, content kind
  (`static-config` | `script`), and (for `script`) a repo-relative
  command
- a `prepare` script under `skills/public/critique/scripts/` (or
  charness-CLI-level) reads the adapter, runs declared sections, emits
  a `charness-artifacts/critique/<slug>-packet.{json,md}` pair
- `critique` SKILL.md adds one workflow step ("consume the packet
  before broad repo sampling") and one output-shape field
  (`Packet Consumed: <path|none>`) so the closeout is auditable
- one charness-repo default section (`changed-files-and-owning-surfaces`)
  ships with a deterministic producer so the contract has live coverage
  in charness itself, not only in cautilus
- `validate_skills` / `validate_adapters` extended (or a new
  `validate_critique_adapter.py`) so adapter-declared packet sections
  fail fast on shape errors

**Open decisions** (resolved 2026-05-14 via inline AskUserQuestion):

1. **Schema kind** → Charness 고유 설계. `charness.critique_prepare_packet.v1`
   envelope (sections + ids + content kind), with cautilus's
   `surface_critique_packet.v1` as one informing example only. Forward
   compatible with multiple consumers; release-packaging vocabulary
   stays in cautilus's section content, not in the envelope.

2. **Adapter contribution mechanism** → Both static-config and
   script-runner. Simple sections (changed-files glob, doc-link list)
   declared inline in `.agents/critique-adapter.yaml`; complex sections
   (cautilus-style structural scanners) declared as a repo-relative
   command the prepare runner invokes.

3. **Workflow requirement strength** → Conditional hard-block. When the
   adapter declares ≥1 `packet_sections` entry, critique closeout cannot
   record done without a `Packet Consumed: <path>` field. Repos with no
   adapter or no declared sections are unaffected (opt-in). Subagent
   reviewers consume the packet payload via the parent's prepare run.

4. **Phase scope** → Single PR covering spec + schema + adapter +
   prepare runner + default `changed-files-and-owning-surfaces` section
   + critique SKILL.md wiring + closeout validator + one charness-repo
   dogfood run. Commit-level slicing keeps each step inspectable.

5. **Adapter name** → `.agents/critique-adapter.yaml` (critique-owned).
   `retro-prepare` counterpart, when added later, will live in
   `.agents/retro-adapter.yaml` (already exists for other retro
   contracts) under its own `packet_sections` slot. Two skills, two
   adapters; DRY only if one shape genuinely fits both.

**Sequencing direction** (subject to user steer on Open Decisions):

1. spec slice — author the schema, adapter shape, workflow change,
   acceptance checks; run a `spec critique` subagent to lock it
2. impl slice 1 — schema + adapter + `prepare` script + default
   `changed-files-and-owning-surfaces` section + validators
3. impl slice 2 — wire `critique` SKILL.md (workflow step + output
   field + closeout validator); dogfood with one real critique run
4. cautilus follow-up — coordinate with cautilus to migrate
   `prepare-surface-critique-packet.mjs` to the new contract (separate
   repo, separate PR; charness ships the contract surface only)

**Autonomous vs pause**: paused 2026-05-13 for user discussion;
resolved 2026-05-14 via inline AskUserQuestion (all five recommendations
selected). Continuing into spec/impl with the resolved boundary.
