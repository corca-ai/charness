## Waste

- We still treated adjacent surfaces as if they guaranteed each other: source checkout vs managed checkout vs installed PATH binary vs checked-in plugin export vs host cache. The repo now has better helpers for this, but the working habit still lagged behind the architecture.
- We kept discovering predictable misses only after broad validation.
- New public skill `premortem` was added without updating `docs/public-skill-validation.json`.
- `quality` and `spec` crossed the `SKILL.md` concise-length gate and only `validate-skills` forced the trim.
- Plugin export needed one more sync after markdown/fence cleanup, and `check-markdown` caught the stale checked-in copy.
- `#24` was closed before the full validation loop had actually gone green, which forced a reopen/close correction cycle.
- We are still slightly too willing to interpret a passing targeted subset as "basically done" when the changed surface clearly implies one more repo-wide contract check.
- The repo's new helper seams are good, but we still leave some of the invocation ordering in human memory. The late misses were mostly sequencing misses, not missing-tool misses.

## Critical Decisions

- We chose to keep real-host support-tool proof as `release`-time human checklist rather than standing CI. That remains the right tradeoff because PATH, package-manager, and host-cache reality still need human smoke on a real machine.
- We chose to promote `premortem` into a standalone public skill instead of copying premortem prose into `spec`, `impl`, and `release` independently. That reduces drift and makes counterweight triage reusable.
- We chose to keep dual-implementation parity detection advisory and weak-heuristic rather than claiming a strong automated detector. This is correct: the helper should say when it sees something, but should not pretend that "no hit" proves absence.

## Trends vs Last Retro

- Compared with the earlier weekly retro written the same day, the broad pattern sharpened: the dominant waste is no longer only wrong assumptions about install/update/support surfaces. It is broader multi-surface sequencing error.
- The repo is better than before at catching drift mechanically, but we still invoke the right mechanical checks too late when a slice touches public-skill policy, checked-in plugin export, and standing markdown/skill gates at once.

## Expert Counterfactuals

- Gary Klein: run the premortem one step earlier, before assuming the slice is ready for issue close or final summary. The changed question would have been: "If this slice still fails after I think it is done, which repo-owned contract did I forget to refresh or reclassify?"
- Atul Gawande: use a short end-of-slice checklist for known omission-prone seams instead of relying on memory. The recurring misses were checklist-class misses: policy table update, plugin sync, concise-gate rerun, issue close after green not before green.
- John Ousterhout: where we keep hitting the same sequence mistakes, reduce the number of independent surfaces or make the closeout driver name them explicitly. Human recall should not be the only thing connecting a new public skill to public-skill validation policy.
