## Waste

- **`MAX_SKILL_MD_LINES=200` 직격을 두 번 맞았다.** 1차 슬라이스에서 HITL
  SKILL.md(208줄) → impl SKILL.md(202줄) → issue SKILL.md(216줄) 순으로 budget을
  넘겼고, 2차 슬라이스에서도 issue SKILL.md(214줄)가 또 넘었다. 매번 검증→압축→
  재검증 사이클을 돌았다. 추가 분량을 알고 있을 때 _먼저_ 기존 텍스트를 압축해
  슬롯을 확보한 뒤 새 contract를 추가하는 순서였다면 한 번에 끝났다.
- **인라인 코드 스팬이 markdown lint에서 wrap 검출에 걸렸다.** 두 슬라이스 모두
  `gh issue create --repo <org/repo>` 같은 긴 backtick span이 줄 끝에 걸쳐
  pre-commit이 한 번 더 실패했다. 본문 작성 시 backtick span 시작을 줄 머리 가까이
  두는 mental model이 없었다.
- **테스트 환경 변수 PATH 설정 미스.** `test_issue_preflight_resolves_adapter_backend_when_gh_absent`에서
  `env={"PATH": f"{bin_dir}"}`로 시작했더니 `python3` 자체가 PATH에 없어 SubprocessError로 죽었다.
  즉시 `:/usr/bin:/bin`을 추가했지만 테스트 작성 시 흔히 잊는 디테일.
- **Bundle Anyway 항목을 사후에야 발견.** premortem이 "closeout-discipline.md를
  SKILL.md 세 anchor에서 cite 했는지 검증하는 cheap test 추가"를 짚어줬는데,
  이는 첫 contract test 작성 시 같이 들어갔어야 하는 패턴이었다.

## Critical Decisions

- **6개를 두 슬라이스로 묶었다.** #122/#124가 sibling (외부 capability proof
  contract), #123 독립, #125/#126/#127가 sibling (`issue` 스킬 closeout/target/
  source identity). 각 슬라이스마다 단일 premortem으로 처리. 모두 한 슬라이스로
  묶었으면 commit message가 닫는 6개 이슈를 모두 surface하기 어려웠을 것.
- **Cautilus disabled 동안 deterministic gate로 closeout한다는 정책 명시.**
  `external-capability-proof-ladder.md`의 closeout 예시에 직접 사용. commit
  message와 close comment에서 highest executed proof level을 명시 → 다음 세션이
  Cautilus 재가동 시 `agent_choice`/`provider_roundtrip`을 어디서 채워야 할지
  파악 가능.
- **#124 verification ladder를 새 shared reference로 분리.** `impl/references/
  verification-ladder.md`는 이미 "강한 도구 선택"이고, #124는 "실제 증명한 레벨"
  이라 의미가 다르다. 동일 파일에 섹션을 추가하기보다 `external-capability-proof-
  ladder.md`로 분리해 cross-skill 인용이 자연스럽게 한다.
- **`issue` skill에서 Christopher Alexander generative-sequence anchor를 채택.**
  사용자 요청이긴 하나, named-person anchor 규칙 (`create-skill` references/
  portable-authoring.md) 안에서 정당한 reasoning frame anchor였다.

## Expert Counterfactuals

- **Donald Knuth (literate programming)**: SKILL.md를 prose-shape 동사 단위로
  쪼개는 사고법이 있었다면 backtick span의 wrap 위치를 라인 시작 근처로 의도적
  배치했을 것. 200줄 budget 압박 안에서도 lint-친화적 wrap을 한 번에 잡을 수
  있다.
- **Christopher Alexander (generative sequence)**: 본인이 직접 인용된 anchor
  답게, 이번 세션 같은 다중 이슈 resolve도 generative sequence로 봤다면 "한
  슬라이스가 다음 슬라이스의 검증 가능 면적을 어떻게 늘리는가"가 명시 결정
  포인트였다. 1차 슬라이스에 verification ladder가 들어갔기 때문에 2차 슬라이스
  의 close comment에서 proof level 표현을 그대로 재사용할 수 있었다 — 이건
  의도된 generative sequence였고, 다음 deferred 작업도 같은 결로 순서를 짠다.
