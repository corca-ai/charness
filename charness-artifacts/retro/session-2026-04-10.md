# Session Retro

## Context

- Unit under review: adding repo-adapter support to `impl` so self-verification
  discovery becomes explicit instead of accidental.
- Why it mattered: a recent downstream report change closed too early because
  `impl` did not force an early search for the strongest repo-available proof
  path.
- Evidence used: the `impl` skill diff, adapter/survey script outputs against
  `charness` and `crill`, the broken `agent-browser` symlink surfaced by the
  survey, and the targeted quality-gate checks run in this session.

## Waste

- Verification capability discovery was previously implicit. That left too much
  to the model's habits, so a UI/report slice could pass string tests while
  skipping a real rendered check.
- The old workflow made missing tooling look like operator forgetfulness rather
  than a first-class repo contract gap. We only learned that `agent-browser`
  was broken after asking the new survey directly.
- Repo-wide eval closeout is still partially noisy on this machine because
  `python3` lacks `jsonschema`. That is separate from this change, but it
  reduced the signal from one broad `pytest` pass.

## Critical Decisions

- Added an `impl` adapter instead of hardcoding repo-specific guidance into the
  shared skill body.
  This kept the public skill portable while giving repos a place to declare
  preferred verification tools and install proposals.
- Added a dedicated verification survey script instead of relying on prose
  alone.
  That turned "look for stronger proof" into an executable bootstrap step and
  exposed broken skill installation state directly.
- Added self-validation hooks in `check-skill-contracts`, `run-evals.py`, and
  `tests/test_quality_gates.py`.
  That constrained future refactors so the new adapter/survey surface is less
  likely to drift silently.

## Expert Counterfactuals

- John Ousterhout would likely have forced the verification preference boundary
  out of the main `impl` prose earlier. The adapter is the simpler interface:
  shared policy in the skill, repo-specific truth in one narrow config seam.
- Charity Majors would likely have asked for the observable signal first:
  "How does the agent know which stronger proof path is available right now?"
  The survey script is the missing observability layer, and the broken
  `agent-browser` symlink is exactly the kind of operational fact that should
  have surfaced before closeout claims.

## Next Improvements

- `workflow`: when a slice touches UI, browser paths, screenshots, static
  reports, or live agents, run the impl verification survey during bootstrap
  and mention the strongest available proof path before implementation starts.
- `capability`: repair or reinstall the broken `agent-browser` skill path so
  repos that prefer browser verification can actually use the stronger proof
  path the adapter now recommends.
- `memory`: if more repos need the same pattern, add a portable preset or
  onboarding note that seeds `impl-adapter.yaml` with common `cmd:`/`skill:`
  verification entries.
