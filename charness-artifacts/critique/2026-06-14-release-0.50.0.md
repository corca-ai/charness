# Release Critique — charness 0.50.0

Scope: short release-hygiene critique for the v0.50.0 minor bump shipping the
#367 `quality` surfaces (CI-recoverability triage lens + `command_timing_log`
ingest). The substantive code critique is already covered by two fresh-eye SHIP
reviews, the rung-2 disposition review, and the #367 resolution critique
(`charness-artifacts/critique/2026-06-14-367-resolution-critique.md`); this pass
covers only the release-specific risks the version-policy / closeout-critique
gate names: bump level, generated surfaces, publish boundary, operator risk.

## Bump level

**Minor (0.49.0 → 0.50.0) is the lightest honest bump.** The release adds new
maintained capability — a new advisory inventory command and a new optional
adapter key — with no compatibility or invocation break: both surfaces are
inert-by-default, add no blocking gate (Floor-Addition Restraint), and change no
existing command's contract. Patch would understate a new operator surface;
major is unjustified (nothing breaks).

## Generated surfaces

The plugin mirror (`plugins/charness/**`), marketplace manifests, and the SLOC
current-pointer were synced and committed during the #367 carrier push; the
publish helper re-syncs install surfaces after the bump and the local quality
gate (77 passed, 0 failed) confirms no stale generated surface. The
`update_instructions` staleness guard is cleared (a 0.50.0 entry was added before
this critique, per the prep step).

## Publish boundary

charness couples version + push + tag + GitHub release through
`publish_release.py`; tag-triggered async publication means tag-push is not
public-verified completion. The closeout must distinguish local/tag state from
the verified public release surface and run the declared fresh-checkout probes
(`charness --help`, `goal check --help`, `doctor`) before claiming done. No
real-host human smoke is required for this slice (`check_real_host_proof`: no
trigger matched).

## Operator risk

Low. Operators refresh via `charness update` (the adapter's
`post_publish_install_refresh`, auto-run after a verified publish). The new
surfaces never run unless invoked (`inventory_ci_recoverable_gates.py`) or
configured (`command_timing_log`), so an operator who updates and changes nothing
sees identical behavior. #367 is already CLOSED by the carrier push, so this
release does not need `--close-issue`. Follow-up #368 is tracked, not shipped.

## Verdict

**Proceed** with the minor bump and publish. Open boundary to respect at
closeout: public release surface verification (tag-triggered workflow) and the
fresh-checkout probes — do not report `public release verified` from tag-push
alone.
