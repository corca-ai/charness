## Failure Angles

- Gawande / operational-checklist (publish sequence completeness + order):
  `publish_release_execute.py::_publish_and_finalize` orders the irreversible
  steps `push → create_release → release_verify → distinct_channel_verification
  → issue_closeout → install_refresh`. Issue closeout runs ONLY after the
  release is created, the post-create verify passes, and a rung-2 distinct-
  channel observer (HTTP fetch of the public release URL, distinct from
  `gh release view`) records a verdict; a rung-1 presence floor
  (`evaluate_release_distinct_channel`) refuses to advance to closeout if that
  verdict is silent. This is the correct ordering for the irreversible boundary —
  #405 is not closed on a terminal-green or a single-channel re-read.
- Minto / communication (will release notes mislead operators about scope):
  the honest scope is "two additive doctrine bullets + a shared review note +
  inventory refresh." Risk: if the generated release notes lean on the carrier
  commit's rich `Resolves #405` JTBD/Boundary ledger, an operator could read
  more than additive prose into it. The Boundary line explicitly scopes out
  "no new top-level lens, no other files, no version bump, no reproduction of
  the downstream incident," which keeps it honest, but release-note framing
  should foreground "doctrine prose, no behavior change" so operators do not
  expect a runtime fix.
- publish-safety / surface-lock (anything consumer-visible or irreversible
  mishandled, incl. closing #405): the issue close is idempotent and verified —
  `ensure_release_issues_closed` checks state first, only closes if not already
  `CLOSED`, then re-reads and `raise SystemExit` if it is still not `CLOSED`.
  So the `Closes #405` in the commit body (which auto-closes on push) and the
  explicit `--close-issue 405` step do NOT double-close or race: the explicit
  step finds it already closed and confirms, or closes + confirms. The only
  consumer surface is additive prose; no removal/rename/migration.
- Bump-honesty (patch vs minor): `references/version-policy.md` reserves `minor`
  for "new additive public skills or support capabilities" and "new operator-
  facing commands, adapters, or install surfaces." The change adds bullets to an
  EXISTING reference file — no new surface. It matches the `patch` criterion
  "wording or metadata fixes that should propagate to installed users." Patch is
  the lightest honest bump; the rationale states why, satisfying the policy
  Guardrail.
