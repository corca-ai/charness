# v0.7.4 Hotfix Critique

Date: 2026-05-19

## Target

Post-release repair for `v0.7.3` issue-closeout verifier and release record
behavior before publishing `v0.7.4`.

## Fresh-Eye Satisfaction

parent-delegated

## Packet Consumed

`charness-artifacts/critique/2026-05-19-063040-packet.md`

## Findings

### Act Before Ship

- `verify-closeout --expect-state OPEN` could report `status: verified`,
  contradicting the issue closeout contract that final verification means
  GitHub state `CLOSED`.
- The default `gh` issue-view path ignored requested `json_fields`, so manual
  fallback verification could not request `comments` and prove the posted
  closeout comment.
- The release helper left the committed release record in a pre-publication
  state after `gh release create` when no issue closeout was requested.

### Bundle Anyway

- Adapter-backed issue `view` commands should prove they carry `{repo}` and
  `{number}` into the selected backend call.
- Proof pointers moved from the broad issue test file to the focused
  closeout-verifier test file and should stay aligned.

### Over-Worry

- The deterministic closeout verifier should not prove mental-model sibling
  search depth. It owns carrier and ledger shape; semantic RCA quality remains
  in the issue/debug workflow and evaluator-backed checks.

### Valid But Defer

- Commit-range-aware critique prepare packets would make post-commit critique
  less dependent on manual target summaries.

## Applied Decision

Fix the three Act Before Ship items and bundle the adapter placeholder and proof
pointer repairs into the same hotfix. Defer commit-range critique packet support.
