# v0.31.0 release critique (corca-ai/charness)
Date: 2026-06-08

## Decision Under Review
Ship `corca-ai/charness` **v0.31.0** (minor bump from 0.30.1). The push bundles 6 unpushed commits (`origin/main..HEAD`): 4× #337 (a new achieve/retro `Structural follow-up:` **disposition floor** — rung 1e — that fires only when a closeout's cited retro names transferable waste via a `## Sibling Search` trigger, plus a rung-2 reviewer mandate and one shared destination vocabulary), 1× #339 draft goal artifact (inert, artifact-only), and 1× release prep (`update_instructions` refresh for 0.31.0). This is a RELEASE critique — the #337 code was reviewed 4× this session; the focus here is the release surface, not re-litigating the floor logic.

## Failure Angles
- **Bump dishonesty.** Minor could overclaim or underclaim — a patch-sized change dressed as minor, or a compatibility break hiding under a minor label.
- **Generated-surface version drift.** The 4 version surfaces (packaging manifest, both plugin manifests, marketplace) could already be inconsistent or pre-drifted before the publish helper bumps them.
- **Grandfather claim false.** The 0.31.0 `update_instructions` asserts EVERY existing goal is unaffected and only `Created ≥ 2026-06-09` + transferable-waste goals are touched. If the gating logic is wrong, a `charness update` could break an in-flight goal in a consumer repo.
- **New required adapter field.** The floor could need a new adapter knob, forcing consumer-repo migration.
- **#339 draft not inert.** The bundled draft goal could carry pursue-ready status or runtime-affecting behavior.
- **Publish-boundary overclaim.** The entry could assert real-host/live/smoke proof the release does not actually reach.
- **Export skew.** The floor logic ships to two source/export trees; a drift between `skills/public/...` and `plugins/charness/...` would mean consumers run stale logic.

## Counterweight Pass
Every angle is answered by executed evidence rather than assertion. (1) **Bump = minor is correct.** This adds a NEW maintained capability (rung 1e structural-follow-up floor) plus an additive operator-facing surface (the new `update_instructions` OPERATOR NOTE), with no public CLI/API removal and no compatibility break — patch would understate a new capability, major is unjustified with no break. (2) **No version drift.** `current_release.py` reports all four surfaces at `0.30.1` with `drift: []` pre-bump; the publish helper's `bump_version.py --part minor` computes `0.31.0` deterministically. (3) **Grandfather claim is TRUE by construction.** `apply_structural_followup_floor` (goal_artifact_disposition.py:336) gates on `enforced = created is None or created >= STRUCTURAL_FOLLOWUP_RULE_DATE` (`2026-06-09`, disposition_form.py:97) AND `transferable = names_transferable_waste(retro_text)`; it returns early (no blocking) unless BOTH hold. Every existing goal is `Created ≤ 2026-06-08`, so `enforced` is False for all of them — they cannot trip the floor. (4) **No new adapter field** — the floor reads only `Created:` and the bound retro; nothing in `.agents/release-adapter.yaml` gained a required key (the release commit is a single-line `update_instructions` addition). (5) **#339 is inert** — `Status: draft`, activation-gated (`/goal @...`), and `Created: 2026-06-08` (below the rule date), so even a hypothetical pursuit would be grandfathered; its `Structural follow-up: TODO —` line is a scaffold placeholder (`_PLACEHOLDER` → ok, not judged). (6) **Proof honesty** — `check_real_host_proof.py` returns `required: false` (no changed path matches the real-host globs); `fresh_checkout_probes` are declared and run at publish; the entry claims only additive/grandfathered/opt-in, no live/provider/smoke. (7) **No export skew** — `scripts/disposition_form.py`, `skills/public/achieve/scripts/goal_artifact_disposition.py`, and the shared vocab all match their `plugins/charness/...` exports byte-for-byte (`diff -q` clean). 44/44 floor tests pass. No angle survives as a release blocker.
