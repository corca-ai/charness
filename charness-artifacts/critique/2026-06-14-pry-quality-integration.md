subagent `pry-reviewer`) of the slice that (1) wires the `pry` testability

## Decision Under Review

- **Consume pry through a quality inventory script.** New
  `skills/public/quality/scripts/inventory_testability_surface.py` wraps
  `pry map`, modeled on the sibling `inventory_sloc.py` (tokei) and
  `run_dead_code_advisory.py` (vulture) support-binary pattern: detect via
  `PRY_BIN`/`shutil.which`, parse JSON, degrade to `status=degraded` when absent,
  always exit 0. Backlog = the `demand=true` substitution-demand subset.
- **Document the contract in the reference, not SKILL.md.** The welded-boundary
  backlog subsection landed in `references/testability-and-selection.md`.
  `quality/SKILL.md` is at 199/200 lines and the reference's own rule keeps
  language-specific (TS/JS) detail out of SKILL.md, so SKILL.md was deliberately
  not edited; its existing testability anchor already routes to the reference.
- **Fix the two findings with an injection seam.** `run-local-eval-test.mjs`
  `runCodexEvaluation`/`runClaudeEvaluation` gained an injectable
  `spawn = spawnSync` parameter (exported, default preserves production
  behavior); pry reclassifies both subprocess findings welded→seamed. Five native
  tests now exercise the previously-unreachable orchestration (success
  normalization, non-zero exit, timeout).
- **Classification.** pry is a support binary (no support skill), already in
  `integrations/tools/dependencies.json`; the only completion gap was quality
  consumption, now closed. `pry` stays advisory and never fails standing quality.

## Failure Angles

- Classifier gaming: is injecting `spawn` a hollow reclassification rather than a
  real testability gain?
- Advisory-contract breach: could the inventory script or seam make pry fail
  standing quality (non-zero exit / a new blocking floor)?
- Behavior drift: does adding a trailing parameter and `export` to the two
  runners change any production path or the CLI entrypoint?
- Scope creep: were generated mirrors or vendored third-party `.mjs` wrongly
  edited; were non-demand or out-of-scope findings churned?
- Test hollowness: do the new tests assert real behavior or just object shape?

## Counterweight Pass

- Not gaming: the codex/claude boundaries wrap real external agent CLIs whose
  orchestration (preflight, timeout, non-zero status with blocker-kind, JSON
  parse, normalization) was untestable without spawning a real agent. The seam is
  the canonical move named in the reference's Recommended Outcomes 1-2, and the
  five tests reach branches that were previously dead to the suite.
- Advisory contract intact: the script returns 0 on ok/degraded/partial-error
  paths; no new `report["ok"] = False` site or `REQUIRED_*` floor was added
  (Floor-Addition Restraint: no blocking floor introduced — advisory only).
- No behavior drift: the two call sites in `evaluateSurface` use the default, so
  production spawns `spawnSync` exactly as before; the `import.meta.url` CLI guard
  is untouched; exports do not collide.
- Scope clean: only the canonical `scripts/agent-runtime` source was fixed; the
  plugin mirror was regenerated by sync, the vendored `slack-api.mjs` and
  `stryker.config.mjs` (0 demand) were left alone.
- Existing reviewed `quality` dogfood case already asserts "treats affected-test
  selection as structural testability" — this slice strengthens that criterion,
  so no new case and no live Cautilus run is required (ask-before-run;
  planner `next_action: none`).
