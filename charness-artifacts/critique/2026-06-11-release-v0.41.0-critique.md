## Findings

### Act Before Ship

- The final release critique artifact must be tracked before publish; the
  packet alone is not the critique record accepted by the release gate.
  Resolution: this artifact is the final critique record and will be committed
  before `publish_release.py --execute`.
- The successful YouTube browser UI proof needed a durable repo artifact rather
  than only chat/session output.
  Resolution: recorded
  `charness-artifacts/gather/2026-06-11-youtube-browser-ui-live-proof.md`.
- The initial release note incorrectly implied the implementation read directly
  from `#segments-container`.
  Resolution: changed the `0.41.0` update instruction to say the route opens the
  transcript UI and extracts transcript segments from the opened UI/accessibility
  snapshot.
- The release note needed to say that `agent-browser` absence degrades honestly,
  not that update alone guarantees browser extraction everywhere.
  Resolution: added explicit skip/diagnostic wording.
- Publish must run through the repo helper, because the helper owns version
  bump, manifest sync, `latest.md`, fresh-checkout probes, tag push, GitHub
  release creation, and post-publish verification.
  Resolution: next command is
  `python3 skills/public/release/scripts/publish_release.py --repo-root . --part minor --critique-artifact charness-artifacts/critique/2026-06-11-release-v0.41.0-critique.md --execute`
  from a clean tree.

### Bundle Anyway

- Keep the softened issue wording: the issue-resolution workflow now requires
  the read step before design, and the helper fails if backend comments are
  absent.
- Commit the prepare packet with the critique artifact so the consumed packet
  remains reviewable.

### Over-Worry

- `packaging/charness.json`, plugin manifests, and marketplace versions staying
  at `0.40.0` before `publish_release.py --part minor` is expected pre-helper
  state.
- `charness-artifacts/release/latest.md` still naming `v0.40.0` is expected
  before the helper writes the `v0.41.0` release audit record.
- Fresh-checkout probes being configured but not run is pre-helper state; the
  helper runs them and blocks publish if they fail.

### Valid But Defer

- A future deterministic gate proving every issue-resolution closeout consumed
  a `comments_read: true` payload is useful, but not a `v0.41.0` release
  blocker.
