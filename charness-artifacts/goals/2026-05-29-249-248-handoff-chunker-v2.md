## Goal

Make the handoff chunker reason over the live backlog (open issues UNION handoff entries, deduped) and fire on a bare `charness:handoff` invocation; reframe the handoff as a thin closeout-only baton (end-only timing) with `## Next Session` demoted from a synced queue to a curation/sequencing memo. Resolves #249 (input too narrow + trigger too specific) and #248 (stage-script input-flag/stdin inconsistency), and codifies end-only handoff timing across the handoff skill contract, operating-contract, and recent-lessons.

## Non-Goals

- Not a rewrite of `state-selection` / `spill-targets` / `continuation-sequence`;
  #249 is explicit that the issue source + trigger are **additive**.
- Not unlocking broader Cautilus surfaces (claim discovery, optimize,
  review-learning, `evaluate live`, Agent orchestration) — eval-only policy holds.
- No hardcoded provider CLI literal (`gh`) in the chunker; issue access routes
  through the existing `issue` / `find-skills` seams.
- Not building the usage-episodes consumer (#243) — separate, design-heavy.
- Not changing the dumb-hook / skill-owns-routing split from #240.

## Boundaries

- **Issue source is a provider/host axis.** Access via existing `issue` /
  `find-skills` routes; repo + label include/exclude and explicit exclusions
  (operator-reserved issues, host-specific exclusions) live behind the handoff
  adapter. Hosts without an issue tracker fall back to doc-only chunking.
  `[axis: provider/host]`
- **Additive source + trigger**, not a pipeline rewrite. `propose_merges` /
  ranker / draft stages keep their current contracts; the issue source feeds the
  same `HandoffEntry` / `ChunkCandidate` shape before `propose_merges`.
- **End-only timing is a hard boundary on the chunker's writes.** At pickup the
  chunker may write only the derived goal skeleton (a forward artifact); it must
  never write `handoff.md`. The baton is written at closeout only.
- Portable per `docs/conventions/implementation-discipline.md`; host-specific
  behavior in adapter/preset, not the skill body.
- **Edit the in-repo source, not the export.** Skill/reference/contract changes
  (S4, S5) edit `skills/public/handoff/...`, `docs/handoff-chunked-routing.md`,
  and `tests/test_handoff_chunker_trigger.py` as the source of truth; the
  `~/.agents/.../plugins/charness/skills/handoff/...` copy is generated on sync.
  Follow mutate → sync → verify; re-sync the plugin export before validators.
- `chunked_routing_lib.py` is already at 816 lines (recent-lessons trap); slices
  2-4 touch it — hold additions tight or split as an explicit slice, do not
  silently grow it.

## User Acceptance

- From a fresh pickup, a **bare `charness:handoff` invocation** (no
  `docs/handoff.md` mention, no explicit task directive) starts chunked routing.
- The presented generative sequence includes **open tracker issues not in the
  handoff `## Next Session`**, clustered by surface (several issues on one
  file/surface group together).
- A handoff entry that cites an issue id (`#NNN`) is **merged with that issue**,
  not double-counted.
- The `handoff.md` document is written **only at closeout**; a pickup leaves it
  untouched (the derived goal skeleton aside). `## Next Session` reads as a
  curation/sequencing memo, not a synced task queue.
